## P0 致命问题

### P0-1 分享页存储型 XSS（无需认证可达）
- **位置**：`routes/share-routes.js:46, 58-59, 73, 178, 180`
- **问题**：`/s/:id` 公开分享页将 `marked.parse(content)` 结果、`notepad.name`、批注 `decoded`/`textInner` 直接拼入 HTML，未做 HTML 转义；marked 默认不过滤原始 HTML。
- **影响**：任意认证用户在 notepad 名称/正文/批注里植入 `<script>`，分享链接被打开（含管理员）即在受害者浏览器同源执行 JS，可经 `/api/*` 偷取全部 notepad 与 thought 数据。
- **证据**：
  ```js
  let htmlBody = marked.parse(textForMarked);
  // 模板里直接拼接：
  // <h1>${notepad.name}</h1> ... ${htmlBody} ... ${decoded}
  ```
- **修复**：对 `notepad.name`、批注文本做 HTML 转义；用 DOMPurify（jsdom + dompurify）清洗 `marked` 输出；加 CSP 响应头。

### P0-2 WebSocket 完全绕过 PIN 鉴权 + 客户端可伪造广播
- **位置**：`server/websocket.js:5-17, 52-78`；`server.js:289-293`
- **问题**：`verifyClient` 只调 `validateOrigin`，从不校验 `dumbpad_auth` PIN cookie；HTTP 层 `requirePin` 只挂 `/api`，不覆盖 WS upgrade。连入后 `ws.on('message')` 对 `notes_update`/`thoughts_update`/`relations_update`/`notepad_change` 直接 `sendToAll(data, ws)` 转发，无服务端校验。
- **影响**：任意满足 origin 的客户端（含同源 XSS，与 P0-1 形成利用链）可连入并向所有其他客户端广播伪造内容；若前端收到 `notes_update` 后自动回写持久化，升级为服务端数据污染。
- **修复**：`verifyClient` 中解析 `info.req.headers.cookie` 校验 PIN；服务端忽略客户端推送的 `*_update` 消息（这些只应由服务端发出）。

### P0-3 SHARE_SECRET 回退到 PIN + Cookie 即 PIN 明文
- **位置**：`server.js:76`；`routes/auth-routes.js:193, 249, 261`
- **问题**：`SHARE_SECRET = process.env.SHARE_SECRET || PIN || 'dumbpad_default_secret_9988'`；当前 `.env` 未设 SHARE_SECRET 且 PIN=666666，分享 token = HMAC(PIN, id)。同时认证 Cookie 与 Bearer token 的值就是 PIN 本身。
- **影响**：拿到任一分享链接（含 id+token）即可离线爆破 4-10 位数字 PIN（6 位仅 100 万次，秒级完成），随后直接登录；Cookie 经日志/代理泄漏即等同于 PIN 泄漏。
- **修复**：强制独立高熵 SHARE_SECRET（启动时缺失即拒绝启动）；Cookie 改为随机 session id（服务端维护 session→PIN 映射），不再直存 PIN。

### P0-4 Thought relations/meta 路径穿越（本地后端）
- **位置**：`scripts/storage.js:525, 533, 541, 546, 559, 567`
- **问题**：本地后端下 `readThoughtMeta/writeThoughtMeta/deleteThoughtMeta/readRelations/writeRelations/deleteRelations` 用原始 `id` 拼 `path.join(DIR, '${id}.json')`，未走 `safeId`（S3 分支反而做了校验）。
- **影响**：`POST /api/thoughts/:id/relations` 的 body 字段 `targetId` 完全用户可控，传入 `../../package` 即可读写 `data/` 目录外任意 `.json`（覆盖 `package.json`、读 `config/storage-state.json` 等）；URL `:id` 经 `%2F` 解码亦可穿越。
- **修复**：本地分支统一 `safeId(id)`；额外校验 `path.resolve` 结果仍位于目标目录内（防符号链接绕过）。

### P0-5 Note 乐观锁存在 TOCTOU，并发写静默丢失
- **位置**：`routes/note-routes.js:51-104`（POST）、`106-198`（PATCH）
- **问题**：`findNotepadById` 读版本 → 409 判断 → `writeNoteContent` → 再次 `readNotepadsMeta` → `version+1` → `saveNotepadsMeta`，全程无锁。两个并发请求带相同 `baseVersion=N`，当前版本=N，`N > N` 为 false，双双通过 409，后写覆盖前写，前一个内容静默丢失，版本只 +1。
- **影响**：多端/多标签同时编辑同一 notepad 时数据丢失，409 保护形同虚设。
- **修复**：为每个 notepadId 引入写锁（仿 `withThoughtWriteLock`），将"版本检查 + 写文件 + 改 meta + 存 meta"放入同一临界区；或用条件写（CAS）。

### P0-6 rebuildRelations 绕过 relation 写锁，并发写竞争丢数据
- **位置**：`scripts/ai-queue.js:957`（对比 `:760`）
- **问题**：`rebuildRelations` 直接 `await buildRelations(meta)`，未走 `withRelationWriteLock`，而路由层 manual relation 写（`thought-routes.js:182/234`）和 `processThought`（`ai-queue.js:760`）都持同一把锁。
- **影响**：rebuild 期间用户手动改关系，或队列并发处理，`syncReverseRelations` 的 read-modify-write 会丢失更新，relations.json 被覆盖。
- **修复**：`rebuildRelations` 内同样用 `withRelationWriteLock(() => buildRelations(meta))`。

### P0-7 readThoughts 全量加载 + rebuildRelations O(n²) IO
- **位置**：`scripts/ai-queue.js:463, 292, 947` + `scripts/storage.js:419`
- **问题**：`buildRelations` 和 `buildThoughtInsightContext` 都 `await storage.readThoughts()`（legacy 模式一次性 `JSON.parse` 整个 thoughts.json），再对每条 `await storage.readThoughtMeta(thought.id)` 串行 N 次 IO（split/S3 模式 N 次文件/S3 GET）。`rebuildRelations` 对 N 条 thought 各调一次 `buildRelations` → O(n²) IO，且每次重复全量加载。
- **影响**：thoughts 增长到几千条后，单次 rebuild 阻塞事件循环、内存堆积、S3 调用配额爆炸。
- **修复**：批量读 meta（一次 list + 并发分批）；`rebuildRelations` 复用一次加载的 metas 集合，避免每条重建都重新全量读。

### P0-8 AI 处理失败时 meta 永久卡 pending，恢复机制不自动运行
- **位置**：`scripts/ai-queue.js:798-830, 917-943`
- **问题**：`processThought` catch 块再调 `storage.writeThoughtMeta` 写 error 状态；若失败原因正是存储不可用（S3 宕机/网络），这次写也抛错，`drainQueue` 的 `Promise.all` 进 catch 仅记日志，但 `queuedIds` 已在 `splice` 时删除，thought 的 meta 停在 `pending` 永不推进。`recoverStalePendingMeta`（默认 120s）存在但不自动调度，需外部触发。
- **影响**：单次存储抖动导致一批 thought 永久卡死，需人工介入。
- **修复**：`drainQueue` catch 中将失败 job 重新入队或保留 `queuedIds`；`init` 时启动定时器周期性调 `recoverStalePendingMeta`。

### P0-9 AI provider 无重试/退避，429 风暴
- **位置**：`scripts/ai-provider.js:381-408`
- **问题**：`requestJSON` 超时直接 `controller.abort()` 抛错，对 429/5xx 无重试、无指数退避。`extract` 失败让整条 thought 进 error，用户/队列重试又立即打 API。
- **影响**：瞬时 429 立即失败并触发重试风暴；批量 backfill 时极易被 LLM 供应商封禁。
- **修复**：对 429/5xx 加 2-3 次指数退避（尊重 `Retry-After`），区分可重试错误与永久错误。

## P2 次要问题（简列）

- `server.js:510` 启动 IIFE `(async()=>{...})()` 无 `.catch()`，迁移失败变 unhandledRejection。
- `server.js:132` `app.listen` 早于路由注册与 `ensureDataDir`，启动短窗口内请求 404。
- `storage.js:40` `s3NotepadKeyCache` 仅在 `setS3Prefix` 时 clear，长期运行无界增长。
- `auth-routes.js:54` 清理 `loginAttempts` 的 `setInterval` 进程级常驻，无 stop。
- `server/indexing.js:101, 109` 两个 `fs.watch` 未在关闭时 `close()`，句柄泄漏。
- `routes/auth-routes.js:195` Cookie `secure` 与 BASE_URL 绑定，HTTP 部署下永不 secure。
- `routes/auth-routes.js:226-228` `/api/pin-required` 泄漏 PIN 长度，缩减爆破空间。
- `scripts/cors.js:8, 79` 默认 `ALLOWED_ORIGINS='*'` + `credentials:true`。
- `routes/auth-routes.js:69-80` `timingSafeEqual` 长度旁路（与 PIN 长度泄漏叠加）。
- `public/managers/thoughts.js:842-855` `_checkScrollLoad` 每次滚动重算 filtered，未复用缓存。
- `public/service-worker.js:63, 293-304` navigation 超时 900ms 偏短，弱网刷新无变化。

## 已确认有防护（避免重复怀疑）

- Thought 写锁（`withThoughtWriteLock`）实现正确，覆盖 POST/PATCH/DELETE 三个主写路径的 thoughts.json 读-改-写段。
- 本地文件写用 temp + rename 原子替换，并对 Windows EPERM/EBUSY 重试 5 次。
- `s3-service.js` `listObjects` 正确使用 `ContinuationToken` 分页，>1000 对象安全。
- AI provider 未配置时 `extract`/`getEmbedding` 返回空值降级，不硬抛。
- `queueThought` 用 `queuedIds` Set 去重（但 migrate 脚本绕过，见 P1-8）。
- PIN 比较用 `crypto.timingSafeEqual`（存在长度旁路，见 P2）。
- `.env` 未被 git 跟踪、`.dockerignore` 排除 `.env`，未烤入镜像；Docker 以 `node` 非 root 运行，仅 EXPOSE 3000。
- `TRUST_PROXY=false` 默认安全，空 `TRUSTED_PROXY_IPS` 时强制关闭代理信任，rate-limit IP 不可经 X-Forwarded-For 伪造。
- 所有 `/api/*`（除 verify-pin/pin-required/config）均经 `requirePin`，未发现接口遗漏鉴权。
- notepad 路径经 `sanitizeFilename(name)` 兜底，`..` 无法逃逸（但 thought relation 路径未防护，见 P0-4）。
- 前端 XSS：`formatThoughtText` 先 `escapeHtml` 再 linkify，thought 渲染无注入。
- outbox 并发有 `outboxInFlight` 标志；WS 重连检查 `readyState` 避免重复创建；fetch 序列化防竞态；resize 监听器正确移除；命令面板/搜索均有防抖。
