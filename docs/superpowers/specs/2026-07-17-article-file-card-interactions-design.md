# `/file` 文本附件拖动与操作菜单设计

## 目标

修复文章编辑器中普通附件拖动后保留原位置、形成重复链接的问题；同时让普通附件拥有与图片一致的编辑态操作入口，而不影响阅读态的直接下载。

## 根因

普通附件渲染为带 `download` 属性的 Markdown 链接。当前代码只把该链接装饰为 `dumbpad-article-file`，没有接管浏览器的原生链接拖放。链接拖进 Vditor 的 `contenteditable` 时，浏览器会执行复制插入，原链接不会移除。

图片已通过受控的 pointer 拖动移动独占块级节点，并在提交前后比较 Markdown，避免复制或误改正文。普通附件应采用同一安全模型，而不是继续依赖原生 HTML5 拖放。

## 交互与边界

- 编辑模式中，只有独占一个块级段落的普通附件可拖动。拖动显示现有块级落点线，支持自动滚动；落下后通过受控 DOM 移动和 Markdown 等价性校验实现真正移动。
- 附件与普通文本、列表、引用或其他附件混排时不允许拖动。这样不会猜测用户的 Markdown 结构，也不会破坏行内链接。
- 编辑模式单击附件卡片打开锚定菜单，菜单只包含下载和删除两个图标按钮。删除移除该 Markdown 链接并触发正常编辑器保存；下载不修改正文。
- 阅读模式单击附件仍直接下载，不显示编辑菜单。
- 删除文章中的链接只移除正文引用，不立即删除对象存储资源；保持现有“避免孤立资源误删”的安全策略。

## 实现边界

- `public/hybrid-editor.js`：提取或复用图片的块级拖放、落点线和提交校验；新增附件链接的块识别、Markdown 链接识别、菜单及删除逻辑。
- `public/Assets/styles.css`：复用图片菜单的尺寸、焦点和主题语言，为附件菜单增加两按钮布局；不新增复杂动效。
- `test_article_file_command_integration.js`：覆盖附件装饰、原生拖放阻止、菜单行为和删除语义；保留 `/file` 命令与上传测试。

## 验证

1. 独占段落的文本附件拖动后只保留一个链接，且落点正确。
2. 混排附件不可拖动，正文不改变。
3. 编辑态菜单提供下载与删除，删除后正文链接消失并触发保存。
4. 阅读态点击直接下载，既不打开菜单也不进入编辑态。
5. 既有图片拖动、图片菜单和 `/file` 上传回归通过。
