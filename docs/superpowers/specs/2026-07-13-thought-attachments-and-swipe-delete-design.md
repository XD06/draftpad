# Thought 附件与滑动删除交互优化设计

## 背景

DumbPad 的 Thought 已支持在 Quick Add 新建弹层和卡片编辑模式中添加附件，附件数据通过 Thought 的 `attachments` 数组随创建、覆盖更新、离线 outbox 和多端同步链路传递。当前浏览态只展示附件，没有直接追加入口；图片附件也没有应用内预览交互。

Thought 卡片已实现向右滑动删除，但现有反馈主要依赖位于卡片左侧的浅色窄轨和图标。卡片仍可跨过阈值并删除，视觉反馈却不够明显，尤其在移动端快速滑动或浅色背景下。

## 目标

1. 图片附件可以点击并在应用内查看大图。
2. Thought 浏览态提供协调、低调的附件追加入口。
3. 点击已有附件区域也能继续追加附件。
4. 滑动删除过程提供清晰但不过度突兀的状态反馈。
5. 保持现有附件数据格式、4 MB 单文件限制、API、outbox、S3/local 存储和 WebSocket 同步行为兼容。
6. 完成后通过项目全量自动化测试，并对关键移动端交互做浏览器验证。

## 非目标

- 不改变附件的存储格式，不新增独立上传服务。
- 不提高现有 4 MB 单文件限制。
- 不重做 Thought 卡片布局、编辑器、标签、AI 或关系面板。
- 不改变滑动删除目前“跨过阈值后松手即删除、不再弹确认框”的行为。
- 不为视频、音频、PDF 增加应用内播放器或阅读器；非图片附件继续使用现有下载行为。

## 交互设计

### 1. 浏览态追加附件

在 Thought 卡片 footer 中增加一个与现有子任务、AI、关系、编辑按钮同规格的附件按钮。按钮只使用线性回形针图标和简短 tooltip/aria-label，不使用强调色常驻背景，避免抢占主要操作的视觉权重。

点击 footer 附件按钮时，打开该卡片专属的隐藏文件选择器。选择文件后沿用 Quick Add 的文件类型范围和 4 MB 校验规则，将合法附件转换为现有 `dataUrl` 对象格式，合并到当前 Thought 的 `attachments` 数组，并通过现有 overwrite API 与 outbox 机制保存。

已有附件区域可点击其空白区域或明确的轻量“添加附件” affordance 继续追加。点击具体图片用于预览，点击具体非图片文件仍执行下载，二者不能误触发追加文件。

无附件时只显示 footer 附件按钮，不额外占用正文空间；有附件时附件区末尾提供低调的追加入口，与缩略图/文件块协调排列。

### 2. 图片预览

图片附件渲染为可聚焦、可点击的按钮语义元素，并保留懒加载缩略图。点击后打开应用内 lightbox：

- 深色半透明遮罩，图片按视口自适应，保留原始比例。
- 显示文件名、当前位置和总图片数。
- 同一 Thought 有多张图片时，支持上一张/下一张按钮和键盘左右方向键。
- 支持 Esc、点击关闭按钮或点击遮罩空白处关闭。
- 提供“下载原图”按钮，继续使用附件 `dataUrl` 和原文件名。
- 打开后将焦点移入预览；关闭后把焦点还给触发缩略图。
- 预览打开时阻止背景滚动；尊重 `prefers-reduced-motion`。

lightbox 仅浏览当前 Thought 内的图片附件，不把非图片文件加入轮播。

### 3. 滑动删除反馈

保留现有向右滑动手势、距离阈值和删除时序，采用已确认的“卡片警示态”视觉：

- 滑动开始后，卡片左侧逐步显现线性垃圾桶图标。
- 随滑动距离增加，卡片边框逐步转为红色，并出现非常浅的红色背景染色。
- 达到删除阈值后，垃圾桶变为白色图标和实心红色圆角底，显示“松开删除”；卡片警示描边增强。
- 设备支持时，仅在首次跨过阈值时触发一次轻微触感反馈；退回阈值内后允许再次跨过时重新反馈。
- 松手删除时，卡片沿当前滑动方向退出并收起高度；未达到阈值则平滑回弹。
- 深色主题使用独立的透明度和边框混合值，确保可见但不刺眼。

实现上不再复用卡片承担常规内阴影的 `::before` 来同时绘制删除轨道。删除动作层和卡片警示样式应职责分离，避免伪元素状态相互覆盖。动作层必须位于卡片可见滑动区域内，不能依赖容易被滚动容器裁切的负坐标。

## 代码边界

### 纯渲染层

`public/managers/thought-card-renderer.js` 负责：

- footer 附件按钮 HTML。
- 图片附件的可交互标记和附件区域追加入口。
- 保持所有附件字段经过 HTML 属性转义。

对应纯 HTML 输出由 `test_thought_card_renderer.js` 覆盖。

### Thought 协调层

`public/managers/thoughts.js` 负责：

- 卡片附件按钮、附件区追加入口和隐藏 input 的事件绑定。
- 统一文件读取、校验和附件对象构造，避免 Quick Add 与编辑/浏览态继续复制逻辑。
- 乐观更新、overwrite API、失败时 outbox 回退。
- lightbox 的创建、打开、切换、关闭、焦点和键盘协调。
- 滑动状态机、阈值状态、触感反馈和删除退出动画协调。

文件处理的纯逻辑应尽量抽到聚焦的 helper 模块，便于测试；DOM 生命周期仍由 `ThoughtsManager` 协调。

### 样式层

`public/Assets/thoughts.css` 负责：

- footer/附件区追加按钮的协调样式。
- 图片缩略图可点击、focus-visible 状态。
- lightbox 桌面/移动端布局、暗色主题和 reduced-motion。
- 滑动动作层、卡片警示渐变、阈值状态和删除退出动画。

### 后端与数据层

现有 `routes/thought-routes.js` 已允许 POST/PATCH 读写 `attachments`，本次原则上不改 API 或存储层。实现前仍需用回归测试确认 overwrite、outbox 和轻量 Thought 列表没有丢弃附件字段。

## 数据流与失败处理

1. 用户从卡片 footer 或附件区选择文件。
2. 前端逐个校验大小；超限文件提示错误，合法文件继续处理，允许一次选择部分成功。
3. 合法文件转换为现有附件对象并追加到 Thought 本地对象。
4. UI 立即重新渲染，用户可看到新附件。
5. 调用现有 overwrite API。
6. 请求失败时沿用 `enqueueThoughtOverwrite()`/outbox 保存完整 Thought；不撤销用户刚选择的附件，避免离线丢失。
7. 读取文件失败时只跳过失败文件并提示，不影响同批其他文件。

lightbox 不修改数据。图片加载失败时显示文件名和失败提示，并保留下载原图入口。

## 诊断与验证策略

### 滑动删除现状诊断

实现前先建立浏览器反馈循环，在移动视口驱动 pointer 手势并检查：

- 卡片的 `--swipe-x` 与阈值 class 是否按距离变化。
- 删除动作层是否实际进入可视区域、是否被父容器裁切。
- 跨过阈值前后图标、文字、边框和背景状态是否可见。
- 未过阈值松手回弹，过阈值松手删除。

根据当前代码，优先验证以下可证伪假设：

1. 动作反馈依赖负坐标伪元素，并被时间线/滚动容器裁切或遮挡。
2. 同一个 `::before` 同时承担卡片内阴影和滑动轨道，状态切换导致视觉职责冲突。
3. `--swipe-icon-opacity` 在滑动距离较大后才开始上升，快速手势中可见时间太短。
4. 红色透明度和尺寸在浅色、暗色主题下对比度不足。

### 自动化测试

新增或扩展测试覆盖：

- 卡片 footer 附件按钮在有/无附件时均正确渲染。
- 图片附件具备可点击预览所需的数据和无障碍属性。
- 附件区追加入口与具体附件点击目标可区分。
- 文件大小校验、附件对象构造和多文件部分成功。
- 浏览态追加附件保持旧附件、通过 overwrite 保存，失败进入 outbox。
- lightbox 只包含图片、索引切换正确、关闭恢复焦点。
- 滑动距离到视觉状态/阈值状态的映射，以及仅跨阈值触发一次反馈。

### 浏览器验收

- 桌面与移动视口均可从 footer 和附件区追加附件。
- 多图片预览、键盘操作、关闭焦点恢复、下载原图可用。
- 非图片附件下载行为未回归。
- 滑动未过阈值回弹，跨过阈值状态明显，松手删除且列表平滑收起。
- 浅色/深色主题及 reduced-motion 下交互可用。
- Quick Add 和编辑模式原有添加/删除附件流程不回归。

## 兼容与风险控制

- 不迁移旧数据；缺少 `attachments` 的 Thought 继续按空数组处理。
- 复用已有 API 和 outbox，避免新增同步协议。
- 卡片手势忽略规则需纳入新增附件按钮、lightbox 和附件追加入口，防止点击附件时触发卡片展开或滑动。
- 事件监听必须在卡片重渲染和 lightbox 关闭时正确清理。
- 新增前端模块时同步更新 service worker 核心资源清单及 PWA 缓存回归测试。
